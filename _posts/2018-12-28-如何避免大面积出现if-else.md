大量 `if-else` 往往意味着逻辑分支过多、职责耦合、扩展困难。这里记录两个最常见的替代思路。

## 策略模式

当问题是“同一件事有多种算法 / 多种处理方式”时，可以考虑策略模式。

典型特征：

- 有一组可替换的处理逻辑
- 调用方只关心“做什么”，不想关心“怎么做”
- 新增一种分支时，希望尽量不改原有代码

示意：

```java
public interface HandlerStrategy {
    void handle();
}

public class AHandlerStrategy implements HandlerStrategy {
    @Override
    public void handle() {
        System.out.println("handle A");
    }
}

public class BHandlerStrategy implements HandlerStrategy {
    @Override
    public void handle() {
        System.out.println("handle B");
    }
}
```

优点：

- 分支逻辑拆开后更清晰
- 新增策略时更容易扩展
- 更符合开闭原则

## 责任链模式

当问题是“多个处理器按顺序尝试处理，谁满足条件谁处理”时，可以考虑责任链模式。

典型特征：

- 一条请求需要经过多层判断
- 每一层都有机会拦截或继续传递
- 想把长长的判断链拆成多个小处理器

示意：

```java
public abstract class Handler {
    private Handler next;

    public Handler next(Handler next) {
        this.next = next;
        return next;
    }

    public final void handle(String request) {
        if (!doHandle(request) && next != null) {
            next.handle(request);
        }
    }

    protected abstract boolean doHandle(String request);
}
```

优点：

- 每个节点职责更单一
- 顺序清晰，适合审批流、校验链、过滤链
- 便于组合和复用

## 还有呢

除了策略模式和责任链模式，实际开发里还常见这些手段：

- `Map<类型, 处理器>`：适合根据枚举、状态、类型做分发
- 多态：让对象自己决定行为，而不是外部堆 `if-else`
- 状态模式：适合状态驱动的复杂分支
- 规则引擎：适合业务规则特别多、经常变化的场景

## 小结

不是所有 `if-else` 都必须消灭。  
真正需要优化的，是那种“分支越来越多、改一处牵一片、读起来很痛苦”的代码。此时优先考虑把“判断”和“处理”拆开，而不是继续往原方法里追加分支。
